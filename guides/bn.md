# ডুয়াল রিডার স্ট্যান্ডার্ড কীভাবে বাস্তবায়ন করবেন

**বৈজ্ঞানিক গবেষণাপত্র, সফটওয়্যার এবং জ্ঞান ব্যবস্থার জন্য মেশিন-যাচাইযোগ্য দাবি তৈরির সম্পূর্ণ নির্দেশিকা।**

> এটিও দেখুন: [bn.json](bn.json) — এআই সিস্টেম এবং ওয়েব ক্রলারদের জন্য এই নির্দেশিকার JSON-LD কাঠামোবদ্ধ ডেটা সংস্করণ।

---

## ডুয়াল রিডার স্ট্যান্ডার্ড কী?

**ডুয়াল রিডার স্ট্যান্ডার্ড (DRS)** হলো জ্ঞান ব্যবস্থার জন্য একটি যাচাইকরণ আর্কিটেকচার। প্রতিটি দাবি — তা গদ্যে লেখা হোক বা কোডে বাস্তবায়িত — দুটি স্বতন্ত্র পাঠক শ্রেণীর দ্বারা পাঠযোগ্য হতে হবে: **মানুষ** এবং **মেশিন**।

DRS দুটি প্রোটোকল নিয়ে গঠিত:

| প্রোটোকল | ডোমেইন | কার্যকারিতা |
|-----------|---------|-------------|
| **DRP** (ডুয়াল রিডার প্রোটোকল) | টেক্সট / গবেষণাপত্র | ৫-অংশের মিথ্যাকরণ প্রেডিকেটের মাধ্যমে গদ্য দাবিগুলোকে মেশিন-মূল্যায়নযোগ্য করে তোলে |
| **GVP** (ভিত্তিযুক্ত যাচাইকরণ প্রোটোকল) | সফটওয়্যার / কোড | টেস্ট বাইন্ডিং এবং কমিট-পিনড প্রমাণের মাধ্যমে মেশিন-মূল্যায়নযোগ্য দাবিগুলোকে মেশিন-যাচাইকৃত করে তোলে |

### তিন পাঠক

| পাঠক | চ্যানেল | যা পড়ে | বিন্যাস |
|-------|---------|---------|---------|
| **মানুষ** | গদ্য | গবেষণাপত্র | প্রাকৃতিক ভাষা |
| **এআই** | JSON | এআই স্তর | কাঠামোবদ্ধ দাবি রেজিস্ট্রি |
| **CI / টেস্ট রানার** | এক্সিকিউটেবল | টেস্ট বাইন্ডিং | টেস্ট নোড ID + কমিট SHA |

## মিথ্যাকরণ কার্নেল K = (P, O, M, B)

উভয় প্রোটোকলের ভাগ করা ভিত্তি। প্রতিটি Type F (মিথ্যাকরণযোগ্য) দাবি একটি নির্ধারণমূলক প্রেডিকেট বহন করে যা ঠিক দুটি রায়ের একটিতে মূল্যায়িত হয়: **FALSIFIED** অথবা **NOT FALSIFIED**।

| প্রতীক | নাম | JSON ফিল্ড | ভূমিকা |
|--------|-----|-----------|--------|
| **P** | প্রেডিকেট | `FALSIFIED_IF` | যৌক্তিক বাক্য যা, TRUE হলে, দাবিটি মিথ্যা প্রমাণ করে |
| **O** | অপারেন্ড | `WHERE` | প্রেডিকেটের প্রতিটি চলকের টাইপকৃত সংজ্ঞা |
| **M** | পদ্ধতি | `EVALUATION` | সসীম, নির্ধারণমূলক মূল্যায়ন পদ্ধতি |
| **B** | সীমানা | `BOUNDARY` + `CONTEXT` | প্রান্তিক মান শব্দার্থবিদ্যা এবং যৌক্তিকতা |

### উদাহরণ প্রেডিকেট

```json
{
  "FALSIFIED_IF": "EXISTS i IN range(len(result)-1) SUCH THAT result[i] > result[i+1]",
  "WHERE": {
    "result": "list · dimensionless · output of sort(input)"
  },
  "EVALUATION": "Run sort on test vectors; check adjacent pairs; finite",
  "BOUNDARY": "len(result) <= 1 → NOT FALSIFIED (vacuously sorted)",
  "CONTEXT": "Ascending order is the documented contract of sort()"
}
```

এই প্রেডিকেট প্রতিটি ভাষায় প্রতিটি এআই সিস্টেমের কাছে একই অর্থ বহন করে। অনুবাদের প্রয়োজন নেই।

### প্রেডিকেট সীমাবদ্ধতা

- **নির্ধারণবাদ**: ঠিক TRUE বা FALSE-তে মূল্যায়িত হতে হবে
- **সসীমতা**: পরিমাণক কেবল সসীম সেটের উপর প্রযোজ্য
- **আত্ম-রেফারেন্স নেই**: কোনো বৃত্তাকার প্রেডিকেট নির্ভরতা নেই
- **সম্পূর্ণতা**: `FALSIFIED_IF`-এর প্রতিটি চলক `WHERE`-তে সংজ্ঞায়িত, এবং বিপরীতক্রমে

## তিনটি দাবির প্রকার

| প্রকার | নাম | বিবরণ | প্রেডিকেট প্রয়োজন? |
|--------|-----|--------|---------------------|
| **A** | স্বতঃসিদ্ধ | মৌলিক ভিত্তি — নকশাগতভাবে মিথ্যাকরণ-অযোগ্য | না (`null`) |
| **D** | সংজ্ঞা | শর্তমূলক সংজ্ঞা — সত্য-যোগ্য নয় | না (`null`) |
| **F** | মিথ্যাকরণযোগ্য | নির্ধারণমূলক প্রেডিকেটযুক্ত পরীক্ষণযোগ্য দাবি | হ্যাঁ — সম্পূর্ণ K = (P, O, M, B) |

## ছয়টি যাচাইকরণ স্তর

প্রতিটি দাবি একটি `tier` ফিল্ড বহন করে যা ঘোষণা করে কোন ধরনের প্রমাণ এটিকে ভিত্তি দেয়।

### নির্মাণগতভাবে ভিত্তিযুক্ত

| স্তর | প্রযোজ্য | অর্থ |
|------|----------|------|
| `axiom` | Type A | মৌলিক, নকশাগতভাবে মিথ্যাকরণ-অযোগ্য |
| `definition` | Type D | সংজ্ঞামূলক, কোনো প্রেডিকেটের প্রয়োজন নেই |

### বর্তমানে ভিত্তিযুক্ত

| স্তর | প্রযোজ্য | অর্থ |
|------|----------|------|
| `software_tested` | Type F | উত্তীর্ণ টেস্ট দ্বারা অনুশীলিত। `test_bindings` অশূন্য, `verified_against` SHA অ-নাল |
| `formal_proof` | Type F | `n_invalid_steps = 0` সহ ধাপ-সূচীকৃত ডেরিভেশন |
| `analytic` | Type F | আনুষ্ঠানিক ডেরিভেশন ট্রেস বা বিশ্লেষণাত্মক যুক্তি দ্বারা যাচাইকৃত |

### স্পষ্টভাবে ভিত্তিহীন

| স্তর | প্রযোজ্য | অর্থ |
|------|----------|------|
| `empirical_pending` | Type F | সক্রিয় প্লেসহোল্ডার বা মুলতুবি তথ্য। ফাঁক নকশাগতভাবে দৃশ্যমান |

## এআই স্তর — কেন্দ্রীয় নথি

এআই স্তর হলো একটি JSON ডকুমেন্ট যা প্রতিটি গবেষণাপত্র বা সফটওয়্যার সিস্টেমের সাথে থাকে। উভয় প্রোটোকল এটির উপর কাজ করে।

### প্রয়োজনীয় বিভাগসমূহ

| বিভাগ | উদ্দেশ্য |
|-------|----------|
| `_meta` | ডকুমেন্ট প্রকার, স্কিমা সংস্করণ, সেশন, লাইসেন্স |
| `paper_id` | অনন্য শনাক্তকারী |
| `paper_title` | মানব-পাঠযোগ্য শিরোনাম |
| `paper_type` | `law_A`, `derivation_B`, `application_C`, বা `methodology_D` |
| `phase_ready` | ফেজ গেট রায় এবং শর্ত অবস্থা (c1–c6) |
| `claim_registry` | প্রকার, প্রেডিকেট, স্তর, বাইন্ডিংসহ সকল দাবির অ্যারে |
| `placeholder_register` | অমীমাংসিত নির্ভরতার অ্যারে |

### স্কিমা

এআই স্তর স্কিমা এখানে পাওয়া যায়: [`ai-layers/ai-layer-schema.json`](https://github.com/thomasbrennan/Fracttalix/blob/main/ai-layers/ai-layer-schema.json)

| শর্ত | প্রয়োজনীয়তা |
|------|--------------|
| **c1** | এআই স্তর স্কিমা-বৈধ |
| **c2** | সকল মিথ্যাকরণযোগ্য দাবি প্রেডিকেটসহ নিবন্ধিত |
| **c3** | সকল প্রেডিকেট মেশিন-মূল্যায়নযোগ্য |
| **c4** | ক্রস-রেফারেন্স ট্র্যাক করা হয়েছে (প্লেসহোল্ডার রেজিস্টার) |
| **c5** | যাচাইকরণ স্বয়ংসম্পূর্ণ (শুধুমাত্র এআই স্তর, গদ্যের প্রয়োজন নেই) |
| **c6** | সকল প্রেডিকেট অ-শূন্য (নমুনা মিথ্যাকরণ পর্যবেক্ষণ বিদ্যমান) |

## কীভাবে বাস্তবায়ন করবেন

### গবেষণাপত্রের জন্য (DRP)

1. গবেষণাপত্র লিখুন (মানব পাঠকদের জন্য গদ্য চ্যানেল)
2. এআই স্তর JSON ফাইল তৈরি করুন (এআই পাঠকদের জন্য মেশিন চ্যানেল)
3. প্রতিটি দাবিকে A (স্বতঃসিদ্ধ), D (সংজ্ঞা), বা F (মিথ্যাকরণযোগ্য) হিসেবে শ্রেণীবদ্ধ করুন
4. প্রতিটি Type F দাবির জন্য ৫-অংশের মিথ্যাকরণ প্রেডিকেট লিখুন
5. প্রতিটি Type F দাবির জন্য একটি `sample_falsification_observation` অন্তর্ভুক্ত করুন (শূন্যতা সাক্ষী)
6. প্রতিটি দাবিতে যাচাইকরণ স্তর নির্ধারণ করুন
7. `ai-layer-schema.json`-এর বিরুদ্ধে যাচাই করুন
8. ফেজ গেট চালান (c1–c6)

### সফটওয়্যারের জন্য (GVP)

1. সফটওয়্যার কী করার দাবি করে তা গণনা করুন
2. প্রতিটি দাবিকে A (অনুমান), D (সংজ্ঞা), বা F (আচরণগত) হিসেবে শ্রেণীবদ্ধ করুন
3. প্রতিটি Type F দাবির জন্য মিথ্যাকরণ প্রেডিকেট লিখুন
4. প্রতিটি দাবি অনুশীলন করে এমন টেস্ট লিখুন বা চিহ্নিত করুন
5. সম্পূর্ণ যোগ্য টেস্ট নোড ID দিয়ে `test_bindings` পূরণ করুন
6. টেস্ট চালান এবং উত্তীর্ণ কমিট SHA `verified_against`-এ রেকর্ড করুন
7. স্তর নির্ধারণ করুন: টেস্ট থাকলে `software_tested`, না থাকলে `empirical_pending`
8. অপরীক্ষিত দাবিগুলো প্লেসহোল্ডার হিসেবে নিবন্ধন করুন
9. `ai-layer-schema.json`-এর বিরুদ্ধে যাচাই করুন

### ন্যূনতম কার্যকর গ্রহণ

সবচেয়ে ছোট কার্যকর DRS গ্রহণ হলো **একটি Type F দাবি একটি টেস্ট বাইন্ডিংসহ**:

```json
{
  "claim_id": "F-1",
  "type": "F",
  "statement": "sort() returns elements in ascending order",
  "falsification_predicate": {
    "FALSIFIED_IF": "EXISTS i IN range(len(result)-1) SUCH THAT result[i] > result[i+1]",
    "WHERE": {
      "result": "list · dimensionless · output of sort(input)"
    },
    "EVALUATION": "Run sort on test vectors; check adjacent pairs; finite",
    "BOUNDARY": "len(result) <= 1 → NOT FALSIFIED (vacuously sorted)",
    "CONTEXT": "Ascending order is the documented contract of sort()"
  },
  "tier": "software_tested",
  "test_bindings": ["tests/test_sort.py::test_ascending_order"],
  "verified_against": "abc1234"
}
```

একটি দাবি। একটি টেস্ট। একটি SHA। DRS সক্রিয়। মূল্য খরচকে ন্যায্যতা দিলে আরও দাবি যোগ করুন।

## নকশা নীতিমালা

**পপারীয় জ্ঞানতত্ত্ব।** আমরা মিথ্যা প্রমাণ করতে পারি কিন্তু সত্য প্রমাণ করতে পারি না। একটি দাবি যা সমস্ত মিথ্যাকরণ প্রচেষ্টা থেকে টিকে যায় সেটি প্রমাণিত নয় — এটি টিকে গেছে।

**সৎ ফাঁক।** প্লেসহোল্ডার হলো সবচেয়ে গুরুত্বপূর্ণ বৈশিষ্ট্য। যখন একটি দাবি `empirical_pending` হয়, তখন সিস্টেম বলে: "আমরা এটি দাবি করি কিন্তু এখনও যাচাই করিনি।" এটি বিকল্পের চেয়ে কঠোরভাবে বেশি তথ্যপূর্ণ, যেখানে অযাচাইকৃত দাবি যাচাইকৃত দাবি থেকে আলাদা করা যায় না।

**সাবস্ট্রেট স্বাধীনতা।** কার্নেল K = (P, O, M, B) জানে না এটি একটি বৈজ্ঞানিক উপপাদ্য নাকি একটি সফটওয়্যার গ্যারান্টি মূল্যায়ন করছে। ভবিষ্যৎ ডোমেইনগুলো (আইনি, নিয়ন্ত্রক, নীতি) কার্নেল পরিবর্তন না করে নিজস্ব প্রোটোকল যোগ করতে পারে।

**মেশিন সার্বজনীন ভাষা।** কার্নেল যুক্তিবিদ্যা এবং গণিতে লেখা, কোনো মানব ভাষায় নয়। একটি প্রেডিকেট যা বেইজিংয়ে TRUE মূল্যায়ন করে সেটি বোস্টনেও TRUE মূল্যায়ন করে। JSON হলো পরিবহন স্তর। বাইনারি যুক্তি হলো সাবস্ট্রেট।
