# Как внедрить Стандарт двойного чтения

**Полное руководство по созданию машинно-верифицируемых утверждений для научных статей, программного обеспечения и систем знаний.**

> См. также: [ru.json](ru.json) — версия этого руководства в формате JSON-LD для ИИ-систем и поисковых роботов.

---

## Что такое Стандарт двойного чтения?

**Стандарт двойного чтения (DRS)** — это архитектура верификации для систем знаний. Каждое утверждение — будь то в прозаическом тексте или в программном коде — должно быть читаемо двумя независимыми классами читателей: **человеком** и **машиной**.

DRS включает два протокола:

| Протокол | Область | Функция |
|----------|---------|---------|
| **DRP** (Протокол двойного чтения) | Тексты / Статьи | Делает прозаические утверждения машинно-оценимыми с помощью 5-компонентных предикатов фальсификации |
| **GVP** (Протокол обоснованной верификации) | ПО / Код | Делает машинно-оценимые утверждения машинно-верифицированными с помощью тестовых привязок и доказательств, закреплённых за коммитами |

### Три читателя

| Читатель | Канал | Читает | Формат |
|----------|-------|--------|--------|
| **Человек** | Проза | Статью | Естественный язык |
| **ИИ** | JSON | ИИ-слой | Структурированный реестр утверждений |
| **CI / средство запуска тестов** | Исполняемый | Тестовые привязки | Идентификаторы тестовых узлов + SHA коммита |

## Ядро фальсификации K = (P, O, M, B)

Общий фундамент обоих протоколов. Каждое утверждение типа F (фальсифицируемое) содержит детерминистический предикат, который вычисляется ровно в один из двух вердиктов: **FALSIFIED** или **NOT FALSIFIED**.

| Символ | Название | Поле JSON | Роль |
|--------|----------|-----------|------|
| **P** | Предикат | `FALSIFIED_IF` | Логическое высказывание, которое при истинности (TRUE) фальсифицирует утверждение |
| **O** | Операнды | `WHERE` | Типизированные определения каждой переменной в предикате |
| **M** | Механизм | `EVALUATION` | Конечная детерминистическая процедура вычисления |
| **B** | Границы | `BOUNDARY` + `CONTEXT` | Семантика пороговых значений и обоснование |

### Пример предиката

```json
{
  "FALSIFIED_IF": "EXISTS i IN range(len(result)-1) SUCH THAT result[i] > result[i+1]",
  "WHERE": {
    "result": "list · dimensionless · output of sort(input)"
  },
  "EVALUATION": "Run sort on test vectors; check adjacent pairs; finite",
  "BOUNDARY": "len(result) <= 1 → NOT FALSIFIED (vacuously sorted)",
  "CONTEXT": "Ascending order is the documented contract of sort()"
}
```

Этот предикат означает одно и то же для любой ИИ-системы на любом языке. Перевод не требуется.

### Ограничения предикатов

- **Детерминизм**: должен вычисляться ровно в TRUE или FALSE
- **Конечность**: кванторы определены только на конечных множествах
- **Нет самоссылок**: нет циклических зависимостей предикатов
- **Полнота**: каждая переменная в `FALSIFIED_IF` определена в `WHERE`, и наоборот

---

## Три типа утверждений

| Тип | Название | Описание | Требуется предикат? |
|-----|----------|----------|---------------------|
| **A** | Аксиома | Фундаментальные посылки — нефальсифицируемы по конструкции | Нет (`null`) |
| **D** | Определение | Стипулятивные определения — не являются истинностно-значимыми | Нет (`null`) |
| **F** | Фальсифицируемое | Проверяемые утверждения с детерминистическими предикатами | Да — полное K = (P, O, M, B) |

---

## Шесть уровней верификации

Каждое утверждение содержит поле `tier`, объявляющее, каким типом свидетельств оно обосновано.

### Обосновано по конструкции

| Уровень | Применяется к | Значение |
|---------|---------------|----------|
| `axiom` | Тип A | Фундаментальное, нефальсифицируемое по конструкции |
| `definition` | Тип D | Дефиниционное, предикат не требуется |

### Обосновано сейчас

| Уровень | Применяется к | Значение |
|---------|---------------|----------|
| `software_tested` | Тип F | Подтверждено проходящими тестами. `test_bindings` непустой, `verified_against` SHA ненулевой |
| `formal_proof` | Тип F | Пошагово индексированная таблица вывода с `n_invalid_steps = 0` |
| `analytic` | Тип F | Верифицировано формальной трассой вывода или аналитическим аргументом |

### Явно необосновано

| Уровень | Применяется к | Значение |
|---------|---------------|----------|
| `empirical_pending` | Тип F | Активный заполнитель или ожидание данных. Пробел виден по конструкции |

## ИИ-слой — центральный артефакт

ИИ-слой — это JSON-документ, сопровождающий каждую статью или программную систему. Это то, с чем работают оба протокола.

### Обязательные разделы

| Раздел | Назначение |
|--------|------------|
| `_meta` | Тип документа, версия схемы, сессия, лицензия |
| `paper_id` | Уникальный идентификатор |
| `paper_title` | Человекочитаемое название |
| `paper_type` | `law_A`, `derivation_B`, `application_C` или `methodology_D` |
| `phase_ready` | Вердикт фазового шлюза и статус условий (c1–c6) |
| `claim_registry` | Массив всех утверждений с типами, предикатами, уровнями, привязками |
| `placeholder_register` | Массив неразрешённых зависимостей |

### Схема

Схема ИИ-слоя доступна по адресу: [`ai-layers/ai-layer-schema.json`](https://github.com/thomasbrennan/Fracttalix/blob/main/ai-layers/ai-layer-schema.json)

## Фазовый шлюз из шести условий

Статья или программный релиз считается **PHASE-READY**, когда все шесть условий выполнены:

| Условие | Требование |
|---------|------------|
| **c1** | ИИ-слой валиден по схеме |
| **c2** | Все фальсифицируемые утверждения зарегистрированы с предикатами |
| **c3** | Все предикаты машинно-оценимы |
| **c4** | Перекрёстные ссылки отслеживаются (реестр заполнителей) |
| **c5** | Верификация самодостаточна (достаточно ИИ-слоя, проза не требуется) |
| **c6** | Все предикаты невакуумны (существует пример фальсифицирующего наблюдения) |

## Как внедрить

### Для статьи (DRP)

1. Написать статью (прозаический канал для человеческих читателей)
2. Создать JSON-файл ИИ-слоя (машинный канал для ИИ-читателей)
3. Классифицировать каждое утверждение как A (аксиома), D (определение) или F (фальсифицируемое)
4. Написать 5-компонентный предикат фальсификации для каждого утверждения типа F
5. Включить `sample_falsification_observation` для каждого утверждения типа F (свидетель невакуумности)
6. Назначить уровень верификации для каждого утверждения
7. Валидировать по `ai-layer-schema.json`
8. Пройти фазовый шлюз (c1–c6)

### Для программного обеспечения (GVP)

1. Перечислить, что программное обеспечение заявляет делать
2. Классифицировать каждое утверждение как A (допущение), D (определение) или F (поведенческое)
3. Написать предикат фальсификации для каждого утверждения типа F
4. Написать или определить тесты, проверяющие каждое утверждение
5. Заполнить `test_bindings` полными идентификаторами тестовых узлов
6. Запустить тесты и записать SHA успешного коммита в `verified_against`
7. Назначить уровень: `software_tested` если тесты существуют, `empirical_pending` если ещё нет
8. Зарегистрировать непротестированные утверждения как заполнители
9. Валидировать по `ai-layer-schema.json`

### Минимальное жизнеспособное внедрение

Наименьшее полезное внедрение DRS — **одно утверждение типа F с одной тестовой привязкой**:

```json
{
  "claim_id": "F-1",
  "type": "F",
  "statement": "sort() returns elements in ascending order",
  "falsification_predicate": {
    "FALSIFIED_IF": "EXISTS i IN range(len(result)-1) SUCH THAT result[i] > result[i+1]",
    "WHERE": {
      "result": "list · dimensionless · output of sort(input)"
    },
    "EVALUATION": "Run sort on test vectors; check adjacent pairs; finite",
    "BOUNDARY": "len(result) <= 1 → NOT FALSIFIED (vacuously sorted)",
    "CONTEXT": "Ascending order is the documented contract of sort()"
  },
  "tier": "software_tested",
  "test_bindings": ["tests/test_sort.py::test_ascending_order"],
  "verified_against": "abc1234"
}
```

Одно утверждение. Один тест. Один SHA. DRS запущен. Добавляйте утверждения, когда ценность оправдывает затраты.

## Принципы проектирования

**Попперовская эпистемология.** Мы можем фальсифицировать, но не верифицировать. Утверждение, пережившее все попытки фальсификации, не доказано — оно выстояло.

**Честные пробелы.** Заполнитель — самая важная функция. Когда утверждение имеет статус `empirical_pending`, система говорит: «мы это утверждаем, но пока не верифицировали». Это строго информативнее альтернативы, при которой неверифицированные утверждения неотличимы от верифицированных.

**Субстратная независимость.** Ядро K = (P, O, M, B) не знает, оценивает ли оно научную теорему или программную гарантию. Будущие области (юридическая, регуляторная, политическая) могут добавлять собственные протоколы без модификации ядра.

**Машинный лингва франка.** Ядро написано на языке логики и математики, а не на каком-либо человеческом языке. Предикат, вычисляемый в TRUE в Пекине, вычисляется в TRUE в Бостоне. JSON — транспортный уровень. Бинарная логика — субстрат.

## Трёхосевая совместимость

| Ось | Обещание | Механизм |
|-----|----------|----------|
| **Прошлое** | Ничто из уже сделанного не ломается | Версионирование схемы, перечисления только с добавлением, постоянное ядро |
| **Латеральная** | Работает во всех областях, языках, инструментах, ИИ-системах | Субстратно-независимое ядро, строково-типизированные привязки |
| **Будущее** | Всё новое может быть добавлено без перепроектирования | Расширяемость протоколов, расширяемость уровней, аддитивная эволюция схемы |
